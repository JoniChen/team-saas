# Plan: Interview-Ready Upgrade — Team SaaS Backend

## Context

The project is a FastAPI + PostgreSQL backend for a multi-tenant SaaS app (users, organizations, projects). It's a functional MVP (~50-60% production-ready) with 6 endpoints, 2 migrations, and 2 happy-path tests. The goal is to transform it into a polished, production-quality codebase that showcases strong engineering skills at mid-level for backend and full-stack interviews: clean API design, correct data modeling, transaction safety, comprehensive testing, and operational readiness.

## Phase 1 — Fix Foundations

**Goal:** Correct subtle bugs and technical debt before building on top of them.

### 1.1 — Fix UUID type consistency across all models

**Problem:** `User.id` uses `UUID(as_uuid=True)` (PostgreSQL native) but `Organization.id` and `Project.id` use `String` with `str(uuid.uuid4())`. FK columns are also `String`. Works by accident via PostgreSQL coercion but breaks on SQLite in tests and is fragile.

**Files:** `app/models/organization.py`, `app/models/project.py`, `app/models/user.py`

- Change `Organization.id` and `Project.id` to `Column(UUID(as_uuid=True), primary_key=True, default=uuid.uuid4)`
- Change all FK columns to `Column(UUID(as_uuid=True), ForeignKey(...))`
- Add `created_at` and `updated_at` timestamps to all three models (`server_default=func.now()`, `onupdate=func.now()`)

### 1.2 — Write Alembic migration: UUID unification + timestamps

**File:** `alembic/versions/<new_rev>_unify_uuid_types_and_add_timestamps.py`  
`down_revision = 'e534e52da7ec'`

Must be hand-written (not autogenerated) because Alembic doesn't emit `USING` clauses for type changes. The upgrade must:
1. Drop FK constraints `users.organization_id → organizations.id` and `projects.organization_id → organizations.id`
2. Alter columns to UUID using `USING column_name::uuid` PostgreSQL cast
3. Re-add FK constraints
4. Add `created_at` / `updated_at` to all three tables

### 1.3 — Add type hints to route handlers and dependencies

**Files:** `app/api/deps.py`, `app/core/security.py`, `app/api/routes/auth.py`, `app/api/routes/organization.py`, `app/api/routes/project.py`

- `get_db() -> Generator[Session, None, None]`
- `get_current_user(...) -> User`
- All route handler parameters annotated explicitly

### 1.4 — Add proper transaction management in `get_db`

**File:** `app/api/deps.py`

Upgrade to auto-commit on success, rollback on exception:

```python
def get_db() -> Generator[Session, None, None]:
    db = SessionLocal()
    try:
        yield db
        db.commit()
    except Exception:
        db.rollback()
        raise
    finally:
        db.close()
```

As routes are touched in Phase 3, remove the explicit `db.commit()` calls from them and use `db.flush()` where a generated ID is needed mid-transaction.

### 1.5 — Add structured logging + health check

**File to create:** `app/core/logging.py`  
Stdlib only (`logging`, `json`, `contextvars`). `RequestIDMiddleware` injects a UUID per request into a `ContextVar`. JSON formatter reads it for every log line: `{"timestamp": "...", "level": "INFO", "request_id": "...", "message": "..."}`.

**File to modify:** `app/main.py`  
- `app.add_middleware(RequestIDMiddleware)`
- `setup_logging()` at module level
- Add `GET /health → {"status": "ok"}` (liveness)
- Add `GET /health/ready` → checks DB connectivity (readiness probe for containers)

## Phase 4 — Testing Infrastructure

**Goal:** Demonstrate that you test edge cases, not just happy paths.

### 4.1 — Create `tests/conftest.py`

Test isolation via PostgreSQL + per-test transaction rollback:

```python
@pytest.fixture(scope="session")
def engine():
    test_engine = create_engine(settings.TEST_DATABASE_URL)
    Base.metadata.create_all(test_engine)
    yield test_engine
    Base.metadata.drop_all(test_engine)

@pytest.fixture(scope="function")
def db(engine):
    connection = engine.connect()
    transaction = connection.begin()
    session = Session(bind=connection)
    connection.begin_nested()  # savepoint for rollback
    yield session
    session.close()
    transaction.rollback()
    connection.close()

@pytest.fixture(scope="function")
def client(db):
    app.dependency_overrides[get_db] = lambda: (yield db)
    yield TestClient(app)
    app.dependency_overrides.clear()
```

Add `TEST_DATABASE_URL: str = ""` to `app/core/config.py`.

### 4.2 — Create `tests/factories.py`

Thin factory helpers (no new packages):
- `make_user(db, email, password, org_id) → User`
- `make_org(db, name) → Organization`
- `make_project(db, org_id, name) → Project`

Use `db.flush()` (not `db.commit()`) so factories participate in the test's rollback transaction.

```yaml
services:
  db:
    image: postgres:16-alpine
    healthcheck: pg_isready
  app:
    build: .
    depends_on: {db: {condition: service_healthy}}
    command: sh -c "alembic upgrade head && uvicorn app.main:app --host 0.0.0.0 --port 8000"
```

Use `service_healthy` condition so the app waits for PostgreSQL before running migrations.

## Interview Talking Points

- **Data modeling**: "I found a UUID type mismatch that worked by accident in Postgres but would've broken in tests — I wrote a hand-crafted migration with USING casts to fix it safely"
- **Transactions**: "Routes don't call commit — the `get_db` dependency auto-commits on success and rolls back on any exception, so you can't accidentally leave partial state"
- **Architecture**: "I separated business logic into a service layer so routes stay thin and the logic is testable without HTTP"
- **Auth**: "I implemented JWT access tokens plus refresh tokens; here's why the expiry window matters and what refresh token rotation protects against"
- **Testing**: "I use a nested transaction savepoint approach per test — every test rolls back to a known state without dropping and recreating tables, so the suite runs fast"
- **CI**: "Every PR runs linting, type checks, and tests — I set this up in Phase 2 so all subsequent work was validated automatically"
- **Docker**: "The app is containerized with a multi-stage build; docker-compose uses `service_healthy` so the app never tries to run migrations before Postgres is ready"
